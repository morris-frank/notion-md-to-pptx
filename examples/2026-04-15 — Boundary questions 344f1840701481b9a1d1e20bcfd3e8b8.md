# Boundary questions

## ‏Material



#### Does the *spatial model* need durable geo-assets and layers, not just farm / field?

Will V2 need first-class reasoning about geometry changes, customer-uploaded arbitrary areas, third-party layers, and non-farm geographies?



If no

- `Farm` + `Field` or equivalent agricultural objects are sufficient for now.

If yes

- Use more general spatial roots such as `AreaAsset` and `SpatialObservationLayer`.

**Evidence needed**

- Sampling app roadmap, remote sensing products, and external GIS / API integration needs.

---

## ‏Material



#### Is a spatial portfolio grouping needed above `AreaAsset`?

Does the platform need to express a customer-owned portfolio of spatial assets independent of any single engagement?



If yes

- Add a grouping concept above `AreaAsset`, such as `Farm`, `Estate`, `Site`, or `Portfolio`.

If no

- Keep spatial assets flat and access them through engagement scope only.

**Evidence needed**

- Multi-site customers, cross-engagement browsing needs, and portfolio-style reporting requirements.

---

## ‏Material



#### Does *sampling campaign* have a formal parent or is it operationally independent?

Is `SamplingCampaign` owned by `ServiceSession` or `InvestigationFrame`, or can it be initiated independently by operations planning?



If owned

- Make the parentage explicit so scientific framing and physical execution stay connected.

If independent

- Accept looser coupling and ensure downstream provenance links remain explicit.

**Evidence needed**

- How campaigns are planned, approved, scheduled, and connected back to findings in practice.

---

## ‏Request



#### Is *commercial container* one lifecycle or several distinct lifecycles?

Do near-term flows share one durable customer container with coherent permissions, pricing, and delivery scope?
*Or* do they split materially across intake, engagement, service session, and entitlement / subscription lifecycles?



If one

- Keep one durable commercial root, such as `Program` or `Engagement`.

If several

- Split intake, commercial relationship, delivery episode, and entitlement into separate lifecycle objects.

**Evidence needed**

- Closed deals, pilot flows, licensing cases, and self-serve / API ambitions.

---

## ‏Request



#### Is *investigation framing* an aggregate root or a per-session frame?

Does the scientific framing have stable identity, independent lifecycle, and reuse across operational sessions?
*Or* is it only session-scoped configuration?



If aggregate root

- Promote `Study` / `InvestigationFrame` to a first-class root with its own identity and lifecycle.

If per-session frame

- Keep `InvestigationFrame` as a child or value object inside `ServiceSession`.

**Evidence needed**

- Real cases where one scientific question spans multiple sessions, years, or geographies.

---

## ‏Request



#### Can one *investigation frame* span multiple service sessions?

Can the same scientific framing be reused across multiple delivery episodes without redefining it each time?



If yes

- Shared framing needs stable identity and should not remain a pure value object.

If no

- Session-local framing is sufficient.

**Evidence needed**

- Monitoring programs, repeated annual studies, and reused protocols over the same areas.

---

## ‏Request



#### Can a *service session* exist before a formal engagement?

Can work begin before a formal commercial container exists?



If no

- Require a parent `Engagement` before opening any `ServiceSession`.

If yes

- Allow `RequestDraft` / `IntakeSession` to act as a lightweight pre-commercial root and define promotion rules clearly.

**Evidence needed**

- Internal work, exploratory work, pre-contract pilots, and what happens when in-flight work becomes contracted.

---

## ‏Work



#### Do *execution retries* create materially different operational states?

Do re-extraction, re-assay, compute reruns, corrections, and customer-caused vs internal reruns need distinct operational and financial handling?



If no

- `ExecutionRun` plus immutable measurement output is sufficient for now.

If yes

- Add explicit retry / attempt and cost-tracking concepts such as `ExecutionAttempt`, `MaterialConsumption`, `ImportBatch`, and `CostEvent`.

**Evidence needed**

- Lab failure modes, QC process, rerun policy, and billing consequences.

---

## ‏Evidence



#### Are *benchmark* and *reference reuse* already product-critical?

Will customer-facing outputs depend on governed corpus versions, reusable benchmark snapshots, cross-customer evidence reuse, or external evidence imports in the next 6–12 months?



If no

- Keep benchmark/reference modeling as a bounded context with open edges.

If yes

- Promote governed domain objects such as `CorpusVersion`, `CorpusPolicy`, and `ExternalEvidenceImport`.

**Evidence needed**

- Product roadmap, contractual reuse language, and AI / training plans.
